# 縦長リデザイン ＆ 進行中タスク機能 計画書

作成日：2026-07-21

---

## 0. この計画のゴール

1. **形状を「円」から「縦長の長方形」へ変更する。**
2. タイマーの顔（メイン画面）そのものに、**いま自分（と AI）が並行して進めている作業を、チェックボックス付きで書き込めるブロック**を載せる。
3. 各作業は **作業時間とともに記録** され、既存の記録／Excel・Sheets 連携にそのまま流れる。

> **キーコンセプト：** これは既存の「タスク選択ウィンドウ（`task.html`）」のような *分類* 機能ではない。
> 「AI に要約を投げつつ、自分は資料を書き、別の AI にコードを生成させる」——
> **同時に走っている複数の作業を見失わないための、常駐チェックリスト**である。

---

## 1. なぜ円をやめるのか（形状の必然性）

| 観点 | 円（現状） | 縦長長方形（新） |
|------|-----------|-----------------|
| 情報密度 | 中央 1 点に集約。リスト表示に致命的に不向き | 上＝タイマー / 下＝リスト、と自然にゾーニングできる |
| 可変長リスト | 円周に沿った配置は 2〜3 行が限界 | 縦スクロールで N 件の作業を並べられる |
| 水位アニメ | 円を満たす（美しいが実装が重い） | **下から満ちる縦カラム**の方がむしろ自然な比喩 |
| 操作ボタン | 下辺に円弧配置（幾何計算が必要） | 上部バー or 下部ツールバーに整列でき、増減が容易 |
| デスクトップ常駐 | 省スペースだが機能拡張の余地なし | 幅を細く保てば省スペース性は維持できる |

**結論：** 「複数行の作業リストを常時表示する」という要件は、円では構造的に成立しない。縦長長方形が最適解。

## 2. 現状デザインの「足し算・引き算」

究極のミニマリズム（＝“美しい時計のような佇まい”）という現状の資産は**壊さない**。その上で機能を足す。

### ✅ 残す（現状の強み）
- **ホバーで操作系がフワッと現れるステルス UI**（非操作時は数字だけの美しい佇まい）
- **水位アニメーション**（縦方向に転用＝下から満ちるカラム）
- 極細フォントの大きなタイマー数字＋直下の現在時刻
- テーマ切替（dark / light / sunset）、休憩モード、ポモドーロ
- 半透明グラスモーフィズム、ドラッグ移動、最前面固定

### ➕ 足す
- **進行中タスク・チェックリスト**（メイン画面下部、複数行、スクロール可）
  - 各行：チェックボックス ／ 作業内容テキスト ／ その作業の累積時間
  - インラインで「＋ 作業を追加」
- **行ごとの計測状態**（走っている行＝時間が加算されている、を示すインジケータ）
- **並列計測**（複数行が同時に走れる＝AI マルチタスクの中核。§4 参照）
- タスク完了（チェック）で、その作業を**時間つきでログへ確定**

### ➖ 引く / 作り替える
- **円弧配置の 6 個の 2 次ボタン** → 上部 or 下部の**水平ツールバー**へ整理（ホバー挙動は維持）
- **プログレスリング（SVG 円）** → **縦の進捗スパイン or 上辺の細いリニアバー**へ
- **円形前提のカーソル透過計算**（`main.js` の距離計算）→ 角丸長方形に合わせて簡素化
- **別ウィンドウのタスク選択（`task.html`）** → 役割が新チェックリストへ統合されるため**段階的に廃止**（§7）
- 各種設定パネル（Excel/Sheets・保存先・タイムラプス）→ 縦長に合わせて再配置（中身は流用）

## 3. 画面ゾーニング（縦長長方形）

```
┌───────────────────────────┐  幅 ≈ 260px / 高さ ≈ 460px

│                           │
│         14:30             │  ← ②現在時刻（小）
│        24:12              │  ← ③タイマー（大・極細）
│      ───────────         │  ← ④リニア進捗 or 縦スパイン
│                           │
│  ACTIVE ─────────────     │  ← ⑤セクション見出し
│  ☑ 資料の要約をAIに依頼  05:20│
│  ☐ LP文章を書く      ▸12:48│  ← 走行中（アクセント）
│  ☐ コード生成(Claude) ▸08:03│  ← 走行中（並列）
│  ☐ メール返信         00:00│
│  ＋ 作業を追加             │  ← ⑥インライン追加
│                           │
│  ▶  ⏸  ⟳    ☕ 🍅 📋     │  ← ⑦下部操作（ホバーで出現）
└───────────────────────────┘
```

- **非操作時：** ①⑦は透明。②③④⑤⑥のみ（“美しい時計＋やることリスト”）。
- **ホバー時：** ①⑦がフェードイン。既存の「連動スライドアップ」演出を踏襲。

## 4. データモデル（新規）★ 本計画の核

### 4-1. 進行中タスク（新キー `mt_active_items`）

```jsonc
{
  "id":            "a1b2c3",     // ランダムID
  "text":          "LP制作",      // 作業内容（親）
  "done":          false,         // チェックボックス
  "accumulatedSec": 0,            // 親自身の作業秒（子がある場合は集計値を表示）
  "running":       false,         // 親自身が時間を加算中か
  "lastStartedAt": null,
  "createdAt":     1737450000000,
  "bornSet":       1,             // 作成時のセット連番（持ち越し判定用・§4-4）
  "children": [                   // 子作業（無ければ空 ＝ 単独タスク）
    { "id":"c1", "text":"要約をAIに依頼", "done":true,  "accumulatedSec":320, "running":false, "lastStartedAt":null },
    { "id":"c2", "text":"文章を書く",     "done":false, "accumulatedSec":768, "running":true,  "lastStartedAt":1737460000000 },
    { "id":"c3", "text":"コード生成(AI)", "done":false, "accumulatedSec":483, "running":true,  "lastStartedAt":1737460500000 }
  ]
}
```

### 4-1-b. 親子（グループ）の扱い ★

- **リーフ or 親**：`children` が空なら単独タスク（従来どおり）、要素があれば **親（グループ）**。
- **計測するのは末端のみ**：時間を加算・ログ確定するのは「子」または「子を持たない親」。
  親（グループ）の表示時間・進捗は **子の集計**（`Σ子.accumulatedSec` ／ `完了子数 / 子数`）。
- **チェックの連動**：親をチェック＝全子を完了。全子が完了したら親も自動で完了扱い。
- **持ち越し（§4-4）はグループ単位**：未完了の子が 1 つでも残る親は、その未完了の子ごと次セットへ持ち越す。全子完了の親はアーカイブ。
- 旧タスク機能の「タスク→詳細」2 階層は、この親子モデルに自然に吸収される（→ `task.html` 廃止の後押し）。

### 4-2. 計測モデル：**並列計測（推奨）**

ユーザー要件「AI に作業させつつ自分は別作業」を素直に満たすため、**複数行が同時に `running` になれる**方式を推奨する。

- 行の ▶ を押す → その行が `running=true`、`lastStartedAt=now`
- 毎秒 tick で `running` な全行の `accumulatedSec` を加算
- 行の ⏸ / チェック → その区間を **セッションとしてログ確定**し停止
- 大きなタイマー（上部）は従来どおり **「フォーカス中の 1 本」**（カウントダウン／ポモドーロ）として機能。フォーカス行と背景で走る AI 行が視覚的に区別される

> **MVP フォールバック：** 実装を軽くするなら「単一アクティブ（走るのは常に 1 行、切替時に前行を確定）」から始め、後で並列へ拡張してもよい。ただし本要件の主旨は並列なので、**最終形は並列**とする。

### 4-3. ログ確定（既存 `mt_logs` を流用＝互換維持）

行を停止／完了したら、その区間を既存スキーマのセッションとして push する：

```jsonc
{ "id", "task": <行のtext>, "detail": null,
  "duration": <区間秒>, "startedAt", "endedAt",
  "mode": "countup", "isBreak": false }
```

これにより **記録ウィンドウ・Excel/Sheets 連携・OneDrive 同期は改修なしで動く**（`store.js` の `SYNC_KEYS` に `mt_active_items` を追加するだけ）。

### 4-4. セット間の持ち越し（キャリーオーバー）★

**「1 セット中に終わらなかった作業は、次のセットにも自動で表示される」** を仕様化する。

ここで言う「セット」＝ 一区切りの作業（ポモドーロ 1 本、または タイマー 1 回の開始〜停止）。

#### ルール

| 状態 | セット終了時の挙動 | 次セットでの表示 |
|------|------------------|-----------------|
| **未完了**（`done=false`） | `mt_active_items` に**残す**（消さない） | **自動で再表示**（＝持ち越し） |
| **完了**（`done=true`） | その区間をログ確定し、アクティブから**除外（アーカイブ）** | 表示しない |

- 持ち越された項目は **累積作業時間（`accumulatedSec`）を引き継ぐ**。
  一方で「そのセットで動かした分」は §4-3 のセッションとして `mt_logs` に確定するので、
  記録側では **セット単位の内訳**が、リスト側では **その作業の通算時間**が見える。
- 持ち越し項目は視覚的に区別する（例：行頭に継続マーク `↪`／ノート案なら赤ペンの「つづき」）。
  判定用に項目へ `bornSet`（作成時のセット連番）を持たせ、`bornSet < 現在のセット` なら「持ち越し」と表示。
- 完了項目は既定でアーカイブするが、**セット終了の瞬間は見せ消し（取り消し線）で数秒残し**、
  「終わった実感」を出してからフェードで消す（UX 上の配慮。実装は任意）。

#### 休憩・ポモドーロとの関係

- 休憩フェーズに入る時：走行中の全項目を一時停止し、区間をログ確定。未完了項目はそのまま保持。
- 次の作業セット開始時：保持していた未完了項目を再表示（必要なら走行を再開）。

## 5. 5 つのデザイン案（`縦長リデザイン_モック.html`）

同梱の HTML に 5 案を並べて比較できるようにした。

| # | 名称 | 一言 | 足し算/引き算の性格 |
|---|------|------|--------------------|
| 1 | **ビーカー（水位）** | 全体が試験管。下から満ちる水位＝進捗、その中にリストが浮かぶ | 水位アニメを縦へ *転用*。最も “らしさ” を残す |
| 2 | **ヘッダー＋リスト** | 上＝タイマー帯 / 下＝チェックリスト。最も実務的で明快 | 情報設計を *足す*。円弧ボタンを水平バーへ *引く* |
| 3 | **並列レーン** | 各作業が横レーン＋走行バー。同時進行を可視化 | AI マルチタスクを最も雄弁に語る *足し算* |
| 4 | **ステルス・ミニマル** | ほぼ線だけ。ドット式チェック＋極薄の時間 | 現状の禅的ミニマリズムを最大限 *維持* |
| 5 | **カードスタック** | 各作業が小カード。アクティブカードが浮いて発光 | 触感的・かんばん風の *足し算* |

**推奨：** ベース構造は **② ヘッダー＋リスト**、並列走行の表現に **③ レーン**の走行ドット、署名的演出に **① ビーカー**の縦水位——を統合した折衷案。まず ② で骨格を固め、③① の要素を段階的に載せるのが安全。

> **【決定 2026-07-21】採用デザイン ＝「ノート」案**（`縦長リデザイン_モック20案.html` の 02）。
> 詳細は **`ノートデザイン_仕様.md`** ／ 実装プロトタイプは **`ノートデザイン_プロトタイプ.html`**。
> 骨格は ② ヘッダー＋リストを踏襲しつつ、罫線ノートの世界観で仕上げる。
> 走行中＝**ハイライトマーカー**、完了＝**赤ペンの見せ消し**、持ち越し＝**赤ペンの継続マーク**（§4-4）。

## 5.5 デザインテンプレート（選択式スキン）★

「見た目を丸ごと選べる」テンプレート機構を用意する。**データ（タスク・ログ）は共通**で、
見た目・形状・作業リストの有無だけがテンプレートで変わる。

| テンプレート | 形状 | 作業リスト | 配色 | 位置づけ |
|--------------|------|-----------|------|---------|
| **ノート**（既定） | 縦長長方形 | あり | 紙（ライト） | 新デザインの本命 |
| **黒板** | 縦長長方形 | あり | チョークボード（ダーク） | ノートのダーク版 |
| **クラシック** | 円 | **なし**（タイマーのみ） | 現行の 3 テーマ（dark/light/sunset） | **現在のミニマルデザインを保存**。最小構成で集中したい人向け |

### 仕様

- 設定キー `mt_template`（`'note' | 'chalk' | 'classic'`）を追加し、`SYNC_KEYS` に含める（好みを複数 PC で共有）。
- テンプレート切替時に **① ルート class、② ウィンドウ形状/サイズ、③ 作業リストの表示可否** を切り替える。
  - クラシック → 円（例 240×240）・リスト非表示
  - ノート/黒板 → 縦長長方形（例 260×460）・リスト表示
- **クラシックでもデータは保持**：作業リストを顔に出さないだけで、別テンプレに戻せば同じタスク・記録が見える。
- **テーマとの整理**：テンプレート＝大枠のスキン、テーマ＝その中の配色。
  - ノート＝「紙／黒板」、クラシック＝既存の「dark/light/sunset」。混乱を避けるため、テンプレートごとに選べるテーマを絞る。
- 既存の水位・リング等はクラシック側の資産として残す（円テンプレートでそのまま活きる）。

### 未決

- クラシックにも作業リストを持たせたい要望が出た場合の扱い（円は構造的に不向き。ホバーで別パネル展開なら可能）。
- `mt_template` を SYNC にするか LOCAL（PC ごとの見た目）にするか。

## 7. 既存機能との整合

- **タスク選択ウィンドウ（`task.html` / `task.js`）：** 新チェックリストへ役割統合。①まず新機能を追加 → ②移行を確認 → ③`btn-task` と別ウィンドウを撤去、の順で**段階的に廃止**。既存の `mt_tasks` / `mt_current_task` は移行期間中は温存。

- **Excel/Sheets・OneDrive 同期・タイムラプス：** §4-3 の通りログ互換のため基本改修不要。
